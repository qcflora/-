# 请示

## 写作规则

- **适用场景**：向上级请求指示、批准。
- **行文方向**：上行文。
- **语气要求**：不卑不亢，把决定权交还上级，不替上级预设结论。
- **约束条件**：
  - **一文一事**：本次只提一个请求事项，多件事请分多份。
  - 必给：请示缘由（为何请示、依据、紧迫性）、请求批准或指示的具体事项、唯一的直接上级机关。
  - 请示事项只写"拟……"（拟申请／拟安排／拟同意），不写"决定……"。
  - 不要夹带汇报性内容或无关事项。
  - 请示一般不抄送下级机关（避免下级先于上级知悉），确需告知另行处理。
  - 事实要求：不得编造未提供或未核实的事实、数据、时间、地点、机构名称、人员身份、文件号和政策依据；信息不足时标注"待核实"或保留占位符。
  - 如用户已提供主送单位、发文单位或日期，则据实写入；未提供时保留占位符，不省略对应结构。

## 版式要求

- **标题格式**：2号小标宋体居中，常见格式为"关于[事项]的请示"。
- **主送单位**：标题下空一行顶格书写，原则上只写一个主送机关，末尾用全角冒号。
- **正文格式**：3号仿宋体，首行缩进2字符；背景依据、请示事项、拟办意见要分层写清，不宜多头请示。
- **特殊元素**：
  - 请示事项宜单列成条，审批请求要明确、可批复。
  - 对资金、编制、项目、机构设置、政策调整等需审批事项，应把"请求批什么"写成可直接批示的句子。
  - 附注通常置于成文日期下一行，左空2字加圆括号，注明联系人姓名和联系电话。
- **与其他文种的区别**：请示与报告的最大区别在于请示必须有请求批准或指示的事项，且一文一事；报告只汇报情况，不夹带请示事项。

## 模板

### 标准结构

```markdown
# 关于[事项名称]的请示

## 主送单位
[主送单位]：

## 一、事项背景
[说明背景、依据、现状和提出请示的必要性。]

## 二、请示事项
[明确需要上级审批、指示或协调的具体事项。]

## 三、请示意见
[写明建议方案、拟采取措施或倾向性意见。]

妥否，请批示。

## 落款
[发文单位]
[日期]

## 附注
（联系人：[联系人] 联系电话：[固定电话]，[手机号]）
```

### 结尾用语

- 常用：**"妥否，请批示。"**
- 也可用：**"当否，请批复。"**
- 或：**"以上请示，请予审批。"**
- 如请求批转执行：**"如无不妥，请批转有关部门执行。"**

### 注意事项

- 当报告写（缺请批语）是常见错误。
- 一文多事会导致请示被退回或搁置。
- 在报告里夹带请示事项会造成处理混乱。
- 事后补请示（先斩后奏）不符合行文规范。
- 不要把背景叙述写得过长掩盖请示核心。
- 请示一般应设置附注，用于标注联系人和联系电话。

### 子类型

1. **请求批准型**：请求上级批准某项工作、项目、经费、编制等。
2. **请求指示型**：就某一问题请求上级给予明确指示或政策解释。
3. **请求协调型**：请求上级协调跨部门、跨地区的有关事项。
