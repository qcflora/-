# 核心规则：兜底骨架模板

## 1. 使用场景

本模板在以下情况下启用：
- 用户选择的文种不在法定 15 种公文或常见 7 种正式材料范围内；
- 用户未明确文种，且素材特征不足以匹配任何已知文种；
- 用户明确要求使用「通用模板」「标准格式」或「无特殊格式」；
- 用户使用的文种有独立模板，但该模板文件缺失或损坏。

**启用原则**：兜底模板仅提供最基本的公文骨架，确保成稿结构完整、格式规范。AI 应在生成过程中根据素材特征和文种常识，推断并补充适合的结构要素。

---

## 2. 通用结构

兜底模板的通用结构如下，各部分的取舍和详略应根据文种特征和素材内容灵活调整：

```
标题
├── 主送单位（如有明确行文对象）
├── 正文
│   ├── 开头/缘由（依据、背景、目的）
│   ├── 主体（事实、分析、措施、要求）
│   ├── 结尾（总结、号召、请求、说明）
│   └── 结尾用语（根据推断的文种选择）
├── 附件说明（如有附件）
└── 落款
    ├── 发文单位
    └── 成文日期
```

---

## 3. 各部分详细说明

### 3.1 标题
- **位置**：版心上方居中。
- **字体**：二号小标宋体（Word 导出时）或 Markdown 一级标题 `# `。
- **构成**：发文机关（可省略）+ 事由 + 文种（或材料性质）。
- **示例**：
  - `关于开展XX专项行动的工作方案`
  - `XX市2024年度工作情况综述`
  - `关于XX问题的说明材料`

**推断提示**：
- 如素材中出现「为了……」「根据……」等目的/依据表述，标题应体现行动性（如「关于开展……」）。
- 如素材以回顾总结为主，标题应体现总结性（如「关于……的总结」）。
- 如素材以解释说明为主，标题应体现说明性（如「关于……的说明」）。

---

### 3.2 主送单位
- **有明确对象时**：在标题下空一行，左顶格编排主送单位全称或规范化简称，后加全角冒号。
  - 例：`各省、自治区、直辖市人民政府：`
- **无明确对象时**：可省略主送单位，直接进入正文。
  - 适用于普发性文件、内部材料、公开信息等。
- **推断方法**：从素材中提取「呈」「报」「请批示」「发至」「印发」等关键词判断行文对象。

---

### 3.3 正文

正文是兜底模板的核心，分为三个逻辑板块。AI 应根据素材内容判断各板块的比重和顺序。

#### 板块一：开头/缘由
- **作用**：交代背景、阐明依据、说明目的。
- **常用表述**：
  - 「为深入贯彻落实……」
  - 「根据《……》（文件号）要求」
  - 「近年来，……取得了显著成效，但……」
  - 「现就……有关事项/情况/工作……如下：」
- **素材映射**：将素材中的政策依据、会议决定、领导要求、形势分析等内容归入此板块。

#### 板块二：主体
- **作用**：承载核心内容，是成稿篇幅最大的部分。
- **常见结构（根据文种特征选择）**：

| 材料性质 | 主体结构建议 |
|----------|-------------|
| 工作部署类 | 工作目标 → 重点任务 → 责任分工 → 保障措施 |
| 情况汇报类 | 基本情况 → 主要做法 → 存在问题 → 下一步打算 |
| 调研分析类 | 调研背景 → 现状分析 → 问题梳理 → 对策建议 |
| 总结回顾类 | 总体情况 → 主要成绩 → 经验体会 → 不足与改进 |
| 说明解释类 | 事项概述 → 具体内容 → 相关依据 → 请求/期望 |
| 方案计划类 | 指导思想 → 实施步骤 → 时间安排 → 工作要求 |

- **层次标注**：主体部分应使用标题层级（`一、` → `（一）` → `1.` → `（1）`）进行结构化，避免大段无层次堆砌。

#### 板块三：结尾
- **作用**：收束全文，根据行文目的作出相应表态或请求。
- **常用模式**：
  - **请求式**：「以上……，请审阅/批示/批准。」
  - **告知式**：「特此……通知/报告/函达。」
  - **号召式**：「请各单位认真贯彻执行/高度重视。」
  - **展望式**：「下一步，我单位将继续……」
- **结尾用语**：根据推断的文种特征，从 `style.md` 的结尾用语规范中选择合适的固定用语。

---

### 3.4 附件说明
- **有附件时**：在正文结束后、落款前，左空两字标注。
  ```
  附件：1.XXXX
       2.XXXX
  ```
- **无附件时**：省略此部分。

---

### 3.5 落款
- **发文单位**：素材中如有明确发文单位，使用全称或规范化简称；如无，保留 `[发文单位]` 占位符。
- **成文日期**：素材中如有明确日期，使用阿拉伯数字书写（如 `2024年7月4日`）；如无，保留 `[日期]` 占位符。
- **位置**：右对齐，成文日期右空四字，发文单位以成文日期为准居中。

---

## 4. 根据文种特征推断结构

当文种不明确时，AI 应通过以下特征推断最合适的结构侧重：

### 4.1 特征识别矩阵

| 关键词/特征 | 推断方向 | 结构建议 |
|-------------|----------|----------|
| 「请批示」「请审批」「当否」 | 请示类 | 缘由简洁 → 事项明确 → 结尾请求批示 |
| 「现将……报告如下」「特此报告」 | 报告类 | 全面汇报 → 客观陈述 → 结尾报告用语 |
| 「请遵照执行」「请认真贯彻」 | 通知/决定类 | 依据明确 → 要求具体 → 结尾执行性用语 |
| 「商请」「贵单位」「函达」 | 函类 | 事由说明 → 协商事项 → 结尾函复请求 |
| 「会议认为」「会议指出」「会议要求」 | 纪要类 | 会议信息 → 议定事项分条 → 无固定结尾 |
| 「方案」「计划」「安排」「步骤」 | 方案/计划类 | 指导思想 → 分阶段任务 → 保障措施 |
| 「总结」「回顾」「成效」「经验」 | 总结类 | 总体概况 → 分项成绩 → 经验与不足 |
| 「调研」「考察」「建议」「对策」 | 调研报告类 | 调研背景 → 问题分析 → 对策建议 |

### 4.2 推断流程
```
分析素材关键词
    ↓
匹配上表特征
    ↓
选择最接近的文种方向
    ↓
调用对应结构建议
    ↓
在成稿开头向用户说明推断结果：
「根据素材特征，本文按『XX类』材料结构进行整合，
  如推断有误，请告知具体文种以便调整。」
```

## 5. 兜底模板的限制

### 5.1 不替代专业模板
- 本模板仅提供基础骨架，不能替代法定公文的专业格式要求（如红头、发文字号、签发人、版记等）。
- 当用户后续明确文种后，应提示用户是否基于专业模板重新生成。

### 5.2 不降低真实性要求
- 使用兜底模板时，仍需严格遵守 `policy.md` 的真实性原则和 `doc-type-guardrails.md` 的零幻觉纪律。
- 不得因「模板通用」而放宽对事实、数据、引文的核查标准。

### 5.3 主动提示义务
- 启用兜底模板时，AI 必须在成稿预览时明确告知用户：
  > 「本次整合使用了通用兜底模板，因文种未明确或超出标准文种范围。如您有特定文种要求或单位内部模板，请提供后重新生成，以获得更精准的格式与结构。」

## 6. 快速参考：兜底模板 Markdown 示例

```markdown
---
title: "[事由]的[材料性质]"
doc_type: "[推断文种或'通用材料']"
direction: "[推断行文方向]"
author: "[发文单位]"
date: "[日期]"
task_id: "task_YYYYMMDD_HHMMSS"
version: "1.0"
---

# [发文机关]关于[事由]的[材料性质]

[主送单位：]（如有）

## 一、[开头/缘由部分标题]

（背景、依据、目的）

## 二、[主体部分标题]

### （一）[子项一]

1. 具体内容
2. 具体内容

### （二）[子项二]

（1）具体内容
（2）具体内容

## 三、[结尾部分标题]

（总结、号召、请求、下一步打算）

[结尾用语，如：特此报告。]

附件：1. [附件名称]
     2. [附件名称]

<div align="right">

[发文单位]

[成文日期]

</div>
```

> 本模板为 `official-doc-assembler` Skill 的最后保障机制，确保在任何情况下都能输出结构完整、格式规范的公文成稿。其核心价值不在于精致，而在于「不失底线」——真实、合规、可编辑。
