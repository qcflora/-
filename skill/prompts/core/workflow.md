# 核心规则：文种判断与工作流程

## 1. 文种判断流程

在整合素材前，必须先确定公文文种。文种决定了结构、用语、行文方向和落款格式。**不可在文种未明的情况下直接生成成稿**。

### 1.1 法定公文 15 种（按《党政机关公文处理工作条例》）
判断优先级：**第一梯队**，如素材特征匹配，优先归入此类。

| 文种 | 核心特征 | 常见场景 |
|------|----------|----------|
| 决议 | 经会议讨论通过的重大决策事项 | 党代会、全会、重要会议 |
| 决定 | 对重要事项作出决策和部署 | 表彰、处分、机构调整 |
| 命令（令） | 公布行政法规、嘉奖、任免 | 政府令、嘉奖令 |
| 公报 | 公布重要决定或重大事项 | 会议公报、联合公报 |
| 公告 | 向国内外宣布重要或法定事项 | 招考、法定事项公布 |
| 通告 | 在一定范围公布应当遵守的事项 | 交通管制、市政管理 |
| 意见 | 对重要问题提出见解和处理办法 | 指导性、建议性文件 |
| 通知 | 发布规章、传达事项、任免人员 | 最常用，转发、印发、会议通知 |
| 通报 | 表彰先进、批评错误、传达精神 | 情况通报、事故通报 |
| 报告 | 向上级汇报工作、反映情况 | 工作报告、调研报告、述职报告 |
| 请示 | 向上级请求指示或批准 | 事项请示、经费请示 |
| 批复 | 答复下级请示事项 | 对请示的正式答复 |
| 议案 | 向同级人大提请审议事项 | 政府议案 |
| 函 | 不相隶属机关之间商洽工作 | 商洽函、询问函、答复函、请求批准函 |
| 纪要 | 记载会议主要情况和议定事项 | 会议纪要、座谈会纪要 |

### 1.2 常见正式材料 7 种（非法定但高频）
判断优先级：**第二梯队**，当不匹配法定公文特征时，归入此类。

| 文种 | 核心特征 | 常见场景 |
|------|----------|----------|
| 方案 | 对某项工作作出系统性部署安排 | 实施方案、整改方案、创建方案 |
| 计划/要点 | 对未来工作的预想性安排 | 工作计划、工作要点 |
| 总结 | 对已完成工作的回顾与评价 | 工作总结、年终总结 |
| 讲话稿 | 代表组织或个人在会议上发言 | 领导讲话、致辞、发言稿 |
| 调研报告 | 对某问题的调查研究及建议 | 专题调研报告、考察报告 |
| 信息/简报 | 简明扼要反映情况、动态 | 工作信息、情况简报、专报 |
| 合同/协议 | 平等主体间设立权利义务 | 合作协议、服务合同 |

### 1.3 自定义模板
- 当上述 22 种均不匹配时，询问用户是否有特定模板。
- 如用户无特定模板，启用 `fallback-template.md` 中的兜底骨架。

### 1.4 文种判断交互流程
```
用户上传素材
    ↓
AI 分析素材特征（目的、对象、内容、语气）
    ↓
AI 给出文种判断建议（1-2 个候选）并说明理由
    ↓
用户确认或修正文种
    ↓
如用户未指定，默认进入「询问文种」步骤
```

## 2. 行文方向判断

确定文种后，进一步判断行文方向，这直接影响用语、格式和结构。

| 方向 | 定义 | 典型文种 | 用语特征 |
|------|------|----------|----------|
| **上行** | 下级机关向上级机关行文 | 请示、报告、意见（上行）、议案 | 谦恭、请求性、汇报性，结尾多用「妥否，请批示」「特此报告」 |
| **下行** | 上级机关向下级机关行文 | 命令、决定、通知、通报、批复、意见（下行） | 权威性、指令性，结尾多用「特此通知」「请遵照执行」 |
| **平行** | 不相隶属机关之间相互行文 | 函、部分通知、部分意见 | 协商性、平等性，结尾多用「请予支持」「特此函达」 |
| **公开** | 向社会公众发布 | 公告、公报、通告、部分通报 | 庄重、简明、规范，一般不标注主送单位 |

### 2.1 判断方法
- 从素材中提取「发文对象」信息：
  - 出现「报」「呈」「请批示」→ 上行
  - 出现「印发」「转发」「各有关单位」→ 下行
  - 出现「贵单位」「商请」「函」→ 平行
  - 出现「现予公布」「社会各界」→ 公开
- 如素材中无明确方向线索，应询问用户。

## 3. 保存与命名约定

### 3.1 默认文件名格式
```
YYYYMMDD_文种_标题摘要
```

- `YYYYMMDD`：文件生成日期，如 `20240704`
- `文种`：使用法定或材料简称，如 `通知`、`报告`、`请示`、`方案`
- `标题摘要`：取标题前 8-15 个汉字，去除标点，如 `关于加强安全生产工作的通知` 取 `关于加强安全生产工作`

### 3.2 示例
- `20240704_通知_关于加强安全生产工作`
- `20240704_请示_关于申请专项经费`
- `20240704_报告_关于上半年工作进展`
- `20240704_函_关于协查有关事项`

### 3.3 版本管理
- 首次生成：使用默认文件名。
- 修改迭代：在文件名后追加 `_v2`、`_v3`，或在日期后加 `_rev1`。
- 如用户指定文件名，以用户指定为准，但应提示其遵循规范。

## 4. 七步工作流程

### 步骤一：询问文种
- 向用户确认或建议公文文种。
- 同时确认行文方向（上行/下行/平行/公开）。
- 若用户已明确文种，跳过此步骤。

### 步骤二：收集素材
- 接收用户上传的文本、文档、图片等素材。
- 对素材进行初步分类：政策依据类、事实数据类、意见建议类、背景参考类。
- 对每份素材标注来源、时间、可靠性等级。
- 识别并标记涉密/敏感内容（见 `doc-type-guardrails.md`）。

### 步骤三：AI 整合
- 根据确定的文种和行文方向，调用对应模板骨架。
- 将素材按逻辑顺序填入骨架：
  - 背景/依据 → 现状/问题 → 措施/建议 → 请求/结语
- 补充必要的过渡句和衔接语，保持段落连贯。
- 严格遵循「零幻觉纪律」，对缺失信息保留占位符。

### 步骤四：合规检查
- 调用 `compliance.md` 中的检查规则，对成稿进行全面扫描。
- 标记 error（必须修改）、warning（建议修改）、info（提示）。
- 生成「合规检查报告」附于成稿之后。
- **仅批注提示，不自动修改原文**。

### 步骤五：预览修改
- 向用户展示 Markdown 格式的成稿预览。
- 同时展示合规检查报告中的关键提示。
- 收集用户修改意见：
  - 如为格式调整，直接迭代；
  - 如为事实性修改，提示用户补充素材或确认信息来源。

### 步骤六：导出
- 确认成稿内容无误后，按用户需求导出：
  - **Markdown**：默认格式，用于存档与版本管理。
  - **Word**：通过 docx.js 生成，按文种自动套用字体和版式（见 `layout.md`）。
- 导出文件按命名约定保存。

### 步骤七：完成
- 在文末附加「交付声明」和「素材来源清单」。
- 记录任务标识、生成时间、版本信息。
- 向用户确认交付完成，并提示后续如需修改可基于当前版本迭代。

## 5. 异常流程处理

| 异常场景 | 处理方式 |
|----------|----------|
| 用户未提供任何素材 | 提示用户上传素材，或询问是否需要空白模板 |
| 素材无法识别（如扫描件模糊） | 提示用户重新上传清晰版本，或手动输入关键内容 |
| 文种判断不确定 | 提供 2-3 个候选文种及理由，请用户选择 |
| 合规检查发现严重错误 | 暂停导出，向用户提示错误详情，等待确认 |
| 用户要求跳过合规检查 | **拒绝**，说明合规检查为强制步骤，不可跳过 |
| 素材涉及多文种 | 询问用户是否拆分多份文件，或以某一文种为主 |

---

> 本规则定义了 `official-doc-assembler` Skill 的标准工作流程，所有任务必须按此流程执行，不得省略关键步骤。
