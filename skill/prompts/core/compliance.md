# 核心规则：合规检查规则

## 1. 检查原则

### 1.1 仅批注，不自动修改
合规检查的核心原则是 **「批注提示，人工裁决」**。AI 在检查过程中：
- **不得**擅自修改成稿原文以消除问题。
- **必须**以批注形式明确指出问题位置、类型、严重程度及修改建议。
- 所有批注应在「合规检查报告」中集中呈现，同时在成稿中对应位置保留可追溯的标记（如 HTML 注释或脚注）。

### 1.2 检查时机
合规检查发生在成稿完成后、用户预览前，是 **强制步骤**，不可跳过。具体流程：
```
AI 生成成稿
    ↓
自动执行合规检查（本规则）
    ↓
生成合规检查报告
    ↓
将报告附于成稿之后，供用户预览时参考
    ↓
用户根据批注决定是否修改
```

## 2. 四类检查规则

### a) 零幻觉检查（真实性核查）

**目标**：确保成稿中不存在编造的事实、数据、文件号或政策引用。

| 检查项 | 判定标准 | 严重级别 |
|--------|----------|----------|
| 政策文件号真实性 | 凡出现「国发」「国办发」「X办发」等格式文号，核查素材中是否有原文支撑；无支撑则标记 | **error** |
| 统计数据溯源 | 凡出现精确数字、百分比、金额，核查是否与素材一致；不一致或无来源则标记 | **error** |
| 会议结论核实 | 凡出现「会议决定」「会议批准」「领导批示」等结论性表述，核查素材是否支撑该结论强度；将「讨论」升级为「决定」则标记 | **error** |
| 时间节点确认 | 凡出现具体日期、期限，核查素材中是否明确；将模糊时间具体化则标记 | **error** |
| 人物机构核实 | 凡出现人名、职务、机构名称，核查素材中是否有依据；无依据则标记 | **error** |
| 引文完整性 | 凡引用政策条文或讲话，核查是否断章取义；缺少上下文导致语义改变则标记 | **warning** |
| 推测变结论 | 将「拟」「计划」「可能」等推测性表述改为确定性表述 | **error** |

**批注格式示例**：
```
【零幻觉检查 · error】第3段：「根据国发〔2024〕15号文件」— 素材中未出现该文件号，
请补充来源或删除。建议：若确需引用，请用户提供文件原文。
```

---

### b) 格式合规检查

**目标**：确保成稿符合公文格式规范（见 `style.md` 与 `layout.md`）。

| 检查项 | 判定标准 | 严重级别 |
|--------|----------|----------|
| 标题层级顺序 | 是否存在跳级（如直接用 `1.` 而无 `一、`）、层级混用（如同时用 `1.` 和 `1、`） | **error** |
| 标题标点 | 标题末尾是否误加句号、逗号（问号、叹号除外） | **warning** |
| 字体对应 | 一级标题是否黑体、二级是否楷体（Markdown 中对应 `##` / `###` 标记） | **warning** |
| 结尾用语 | 是否使用符合文种的结尾用语；是否遗漏结尾用语 | **error** |
| 落款格式 | 是否包含发文单位和成文日期；日期是否使用阿拉伯数字；位置是否正确 | **error** |
| 主送机关 | 上行文、平行文是否遗漏主送机关；主送机关格式是否正确 | **error** |
| 首行缩进 | 正文段落是否首行缩进 2 字符 | **warning** |
| 附件说明 | 如有附件，是否在正文后标注附件说明；附件名称是否完整 | **warning** |
| 页边距与版面 | Word 导出时页边距是否符合 A4 标准（上37/下35/左28/右26mm） | **warning** |

**批注格式示例**：
```
【格式合规 · error】结尾用语缺失：本文种为「请示」，但未使用「妥否，请批示。」
或同类结尾用语。请在落款前补充。
```

---

### c) 文种匹配检查

**目标**：确保成稿内容符合所选文种的法定特征，不发生文种误用。

| 检查项 | 判定标准 | 严重级别 |
|--------|----------|----------|
| 请示夹带报告 | 「请示」中是否夹杂大量汇报性内容而请批事项不明确；即「请示报告」混用 | **error** |
| 报告夹带请示 | 「报告」中是否出现「请批准」「请批示」等请批事项；报告不得夹带请示 | **error** |
| 通报与通知混淆 | 「通报」是否用于发布规章或任免人员（应使用通知）；「通知」是否用于表彰批评（应使用通报） | **warning** |
| 函的适用范围 | 「函」是否用于有隶属关系的上下级之间（应使用请示/报告/通知） | **error** |
| 上行文语气 | 上行文（请示、报告）是否使用命令式、祈使式语气，缺乏谦恭态度 | **warning** |
| 下行文语气 | 下行文（通知、通报、决定）是否过于软弱，缺乏权威性和执行力 | **info** |
| 意见的方向性 | 「意见」作为上行文时是否过度指令化；作为下行文时是否缺乏指导性 | **warning** |
| 纪要纪实性 | 「纪要」是否加入过多主观评价或未经会议讨论的结论 | **error** |

**批注格式示例**：
```
【文种匹配 · error】本文种为「报告」，但第5段出现「以上事项，请予批准」。
报告文种不得夹带请批事项。建议：如需批准，另发请示；或删除请批表述，改为「特此报告」。
```

---

### d) 用语规范检查

**目标**：标记口语化、非规范化表达，提示改为标准书面语。

| 检查项 | 判定标准 | 严重级别 |
|--------|----------|----------|
| 口语化表达 | 出现「这个」「那个」「说白了」「挺多的」「一下」等口语词汇 | **warning** |
| 网络流行语 | 出现「点赞」「给力」「硬核」「破防」「内卷」等网络用语 | **error** |
| 模糊限制语 | 出现「大概」「可能」「差不多」「基本上」等缺乏精确性的表述（除非素材原文） | **info** |
| 情绪化用语 | 出现「令人痛心」「非常高兴」「极其恶劣」等强烈主观情绪词（除非素材原文为讲话稿） | **warning** |
| 第一人称随意 | 出现「我们认为」「我方觉得」「我个人看法」等不规范自我指代 | **warning** |
| 方言词汇 | 出现具有地域特征的方言词汇 | **warning** |
| 英文缩写 | 未经解释直接使用英文缩写（如「KPI」「OKR」「AI」首次出现未标注中文） | **info** |
| 数字用法 | 应使用阿拉伯数字处使用汉字（如「二十五」应为「25」，除成文日期和层次序数外） | **info** |

**批注格式示例**：
```
【用语规范 · warning】第2段第3行：「说白了，就是各部门要配合好」— 「说白了」为口语化表达。
建议改为：「本质上，各部门需加强协同配合」。
```

## 3. 严重级别定义

| 级别 | 名称 | 含义 | 处理要求 |
|------|------|------|----------|
| **error** | 错误 | 违反硬性规范，可能导致公文失效、产生歧义或法律风险 | **必须修改**后方可导出；如用户坚持保留，需用户明确确认并记录 |
| **warning** | 警告 | 存在瑕疵或风险，建议修改但不影响公文效力 | 建议修改；如用户不采纳，可导出但需在报告中保留记录 |
| **info** | 提示 | 优化建议或格式提示，不影响核心合规性 | 供用户参考；可忽略 |

---

## 4. 合规检查报告格式

每份成稿必须附带一份「合规检查报告」，置于成稿之后、交付声明之前。

### 4.1 报告结构
```markdown
---

## 合规检查报告

**检查时间**：2024年7月4日 14:30  
**成稿版本**：v1.0  
**检查范围**：零幻觉 / 格式合规 / 文种匹配 / 用语规范

---

### 一、零幻觉检查
| 序号 | 位置 | 问题描述 | 级别 | 建议 |
|------|------|----------|------|------|
| 1 | 第3段 | 编造文件号「国发〔2024〕15号」 | error | 请补充来源或删除 |
| 2 | 第5段 | 将「拟推进」改为「将启动」 | error | 恢复素材原貌或保留占位符 |

### 二、格式合规检查
（同上表格结构）

### 三、文种匹配检查
（同上表格结构）

### 四、用语规范检查
（同上表格结构）

---

**检查结果汇总**
- error：X 项
- warning：Y 项
- info：Z 项

**建议操作**：
> 请先处理 error 级别的问题，确认后处理 warning 级别的问题。info 级别的问题可根据需要选择性优化。
```

### 4.2 无问题报告
若检查未发现任何问题，仍需生成报告：
```markdown
## 合规检查报告

**检查时间**：2024年7月4日 14:30  
**检查结果**：本次检查未发现 error 或 warning 级别的问题。info 级别提示 0 项。

> 合规检查通过，可进入导出环节。
```

## 5. 检查执行与责任

### 5.1 自动化与人工结合
- AI 应在生成成稿后自动执行上述四类检查。
- 对于复杂语境下的语义判断（如是否属于「断章取义」），AI 应给出最可能的标记，并在批注中说明判断依据。
- 最终裁决权归用户所有，AI 不得强制修改。

### 5.2 用户跳过检查的约束
- 如用户明确要求跳过合规检查，AI 应**拒绝执行**，并说明：
  > 「合规检查是本 Skill 的强制步骤，不可跳过。该步骤仅生成批注报告，不会修改您的成稿内容。您可以在预览后选择是否采纳批注建议。」

### 5.3 检查规则的迭代
- 随着使用场景的扩展，本规则可新增检查项或调整判定标准。
- 新增规则不得降低零幻觉检查的严格性。

---

> 本规则为 `official-doc-assembler` Skill 的合规保障机制，是成稿交付前的最后一道质量关卡。检查本身不改变成稿内容，只提供问题清单与修改建议，确保用户对最终内容拥有完全控制权。
